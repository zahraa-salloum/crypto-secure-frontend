How It Works

Understanding stream cipher cryptography and implementation

🔐 What are Stream Ciphers?

Stream ciphers are symmetric key encryption algorithms that encrypt plaintext one bit or byte at a time, combining it with a pseudorandom keystream generated from a secret key.

Basic Operation

Ciphertext = Plaintext ⊕ Keystream

The encryption operation uses XOR (⊕) to combine each bit of plaintext with the corresponding bit from the keystream. Decryption uses the same operation because XOR is self-inverse.

✅ Advantages

  • Fast encryption/decryption
  • Low memory requirements
  • Suitable for real-time streaming
  • No padding needed

⚠️ Challenges

  • Key must never be reused
  • No error propagation
  • Vulnerable to bit-flipping attacks
  • Requires synchronized state

🔑 RC4 Stream Cipher

RC4 (Rivest Cipher 4) was designed by Ron Rivest in 1987 and widely used in protocols like SSL/TLS and WEP. Despite its simplicity and speed, it's now deprecated due to several discovered vulnerabilities.

1

Key-Scheduling Algorithm (KSA)

Initializes a 256-byte state array (S) using the secret key. The key is repeated to fill the array, then permuted based on key bytes. 

for i from 0 to 255:
    S[i] = i
    
j = 0
for i from 0 to 255:
    j = (j + S[i] + key[i mod keyLength]) mod 256
    swap(S[i], S[j])
2

Pseudo-Random Generation Algorithm (PRGA)

Generates the keystream by continuously permuting the state array and outputting bytes based on the permutation. 

i = j = 0
while generatingOutput:
    i = (i + 1) mod 256
    j = (j + S[i]) mod 256
    swap(S[i], S[j])
    K = S[(S[i] + S[j]) mod 256]
    output K
3

Encryption/Decryption

XOR each byte of plaintext/ciphertext with the corresponding keystream byte. 

for each byte in data:
    keystreamByte = PRGA.next()
    output byte ⊕ keystreamByte
🚫 Known Vulnerabilities:
  • Biased keystream output (especially in first bytes)
  • Related-key attacks
  • Distinguishing attacks with ~2^30 samples
  • Weak keys that produce predictable keystreams

📱 A5/1 Stream Cipher

A5/1 is a stream cipher used to encrypt voice data in GSM mobile networks. It uses three Linear Feedback Shift Registers (LFSRs) with irregular clocking controlled by a majority function.

Three LFSR Structure

LFSR-1

Length: 19 bits

Taps: 13, 16, 17, 18

Clock bit: 8

LFSR-2

Length: 22 bits

Taps: 20, 21

Clock bit: 10

LFSR-3

Length: 23 bits

Taps: 7, 20, 21, 22

Clock bit: 10

1

Key Normalization

A5/1 requires exactly 64 bits (8 bytes) as the session key (Kc). All key bytes are XOR-folded into an 8-byte block so that every byte of the input contributes to the full 64-bit key — no material is silently discarded or padded with zeros. 

normalizedKey[8] = {0}
for each byte b in key:
    normalizedKey[b mod 8] ^= b
2

Initialization — Phase 1: Load 64-bit Session Key

All three LFSRs start at zero. Each of the 64 key bits (LSB-first within each byte) is XORed into bit 0 of every register, then all three registers are clocked unconditionally using their feedback taps. 

for i from 0 to 63:
    LFSR1[0] ^= keyBit[i]
    LFSR2[0] ^= keyBit[i]
    LFSR3[0] ^= keyBit[i]
    clock LFSR1, LFSR2, LFSR3  // unconditional
3

Initialization — Phase 2: Load 22-bit Frame Number

A 22-bit frame counter (COUNT) is loaded LSB-first using the same XOR-then-clock procedure. This makes every frame produce a unique keystream even when the session key is reused — the core security property of A5/1 in GSM. 

for i from 0 to 21:
    LFSR1[0] ^= frameBit[i]
    LFSR2[0] ^= frameBit[i]
    LFSR3[0] ^= frameBit[i]
    clock LFSR1, LFSR2, LFSR3  // unconditional
4

Initialization — Phase 3: Warm-up (100 cycles)

100 majority-clocked cycles are run and the output is discarded. This mixes the internal state thoroughly before keystream production begins. 

for i from 0 to 99:
    clockMajority(LFSR1, LFSR2, LFSR3)  // output discarded
5

Majority Clocking

Each LFSR has a dedicated "clock bit". In each keystream cycle the majority value of the three clock bits is computed; only registers whose clock bit matches the majority are advanced. This irregular clocking is the source of A5/1's non-linearity. 

majority = (clockBit1 + clockBit2 + clockBit3) >= 2 ? 1 : 0

if (clockBit1 == majority) clock LFSR-1
if (clockBit2 == majority) clock LFSR-2
if (clockBit3 == majority) clock LFSR-3
6

Keystream Generation

After clocking, the output bit is the XOR of the highest-index bits of all three registers. In real GSM usage this produces a 114-bit burst per frame (one burst per TDMA frame direction). 

keystreamBit = LFSR1[18] ⊕ LFSR2[21] ⊕ LFSR3[22]
🚫 Known Vulnerabilities:
  • Can be broken in real-time with rainbow tables
  • Time-memory tradeoff attacks feasible
  • Correlation attacks on individual LFSRs
  • Weak against known plaintext attacks

✅ Cryptographic Best Practices

🔑 Key Management

  • Never reuse keys or initialization vectors
  • Use cryptographically secure random generators
  • Store keys separately from encrypted data
  • Implement proper key derivation functions

🛡️ Modern Alternatives

  • AES-256-GCM: Authenticated encryption standard
  • ChaCha20-Poly1305: Fast stream cipher with authentication
  • TLS 1.3: Secure transport protocol
  • Signal Protocol: End-to-end messaging encryption

⚠️ Common Pitfalls

  • Rolling your own crypto (use established libraries)
  • Using deprecated algorithms in production
  • Inadequate key length or randomness
  • Encryption without authentication
